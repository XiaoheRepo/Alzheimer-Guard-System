## 1. 执行目标与边界

### 1.1 执行目标

1. 在不偏离契约的前提下，将 Android 从骨架工程推进到可上线状态。
2. 覆盖手册定义的全量页面范围（38 个主页面）并完成后端联调闭环。
3. 建立稳定交付机制：每个阶段都有可演示成果、质量门禁与回滚预案。

### 1.2 当前基线判断

1. 当前仓库仍为单模块初始工程，业务模块尚未落地。
2. 网络、鉴权、缓存、实时通道、错误码映射均需从基座开始建设。
3. 需要先完成架构和契约治理，再进入大规模页面开发。

### 1.3 不在本方案内

1. iOS/Web 客户端实现细节。
2. 后端架构重构工作。
3. 非手册约束的新需求扩展。

---

## 2. 总体策略

1. 先冻结契约再开发：需求、字段、状态机、错误码先对齐。
2. 先搭基座再铺页面：统一网络与状态管理先落地。
3. 先做闭环再扩展：优先完成可运行链路，再做广度覆盖。
4. 每个阶段必须满足 DoD（代码 + 测试 + 联调 + 埋点 + 文档）。

## 4. 阶段交付清单（每阶段强制）

1. 功能交付：对应页面的可运行实现与演示脚本。
2. 契约交付：接口字段映射表、错误码处理矩阵、状态机映射表。
3. 测试交付：
   - 单元测试（UseCase/Mapper/Reducer）
   - 集成测试（Repository + MockWebServer）
   - UI 关键路径测试（登录、线索提交、通知已读、任务刷新）
4. 可观测性交付：trace_id 贯通、WS 生命周期埋点、关键业务指标埋点。
5. 文档交付：接口变更、行为变更、限制与已知问题同步更新。

---

## 5. 后端对齐执行机制

### 5.1 契约冻结机制

1. 每周一契约评审会：确认当周接口变更窗口。
2. 任一字段变更必须先改 API 文档，再建客户端任务。
3. 字段状态仅允许：`已确认`、`待确认`、`有风险`。
4. `待确认` 或 `有风险` 项禁止进入开发完成态。

### 5.2 联调节奏

1. 每日 15 分钟联调站会，优先清理阻断项。
2. 每周一次契约回归日，验证关键接口不漂移。
3. 联调问题单统一模板：请求参数、响应体、trace_id、复现路径、期望行为。

### 5.3 强约束检查点

1. 所有请求必须注入 `X-Trace-Id`。
2. 所有写请求必须注入 `X-Request-Id`。
3. 鉴权接口必须带 `Authorization: Bearer`。
4. 匿名链路必须使用 `X-Anonymous-Token`，且不在 Body 传 `entry_token`。
5. 429 必须处理 `Retry-After`。
6. 401 (`E_GOV_4011`) 必须执行统一会话失效流程。

## 6. 架构与工程实施顺序（执行级）

### 6.1 工程拆分顺序

1. 先从单模块迁移为多模块骨架：`app` + `core-*` + `feature-*`。
2. 优先创建 `core-network`、`core-common`、`core-datastore`、`core-database`、`core-realtime`。
3. 再创建 feature 模块并按阶段接入页面。

### 6.2 基础能力优先级

1. 优先级 P0：统一网络响应、拦截器、错误码映射。
2. 优先级 P1：会话管理、Token 安全存储、401 统一处理。
3. 优先级 P1：分页游标、缓存策略、离线重试队列。
4. 优先级 P2：WS ticket 建链、去重防乱序、断线重连与轮询降级。

## 7. 页面批次实施计划（按业务域）

### 7.1 批次 A（匿名 + 鉴权 + 通知）

页面：PUB-01~05、AUTH-01~03、NOTI-01。

验收要点：
1. 匿名上报成功后仅展示回执，不泄露患者隐私。
2. `E_CLUE_4012` 必须触发清理匿名凭据并引导重新扫码。
3. 登录成功写入会话；登出清理 Token、WS、缓存。
4. 通知“先本地后远端”，单条已读失败回滚。

### 7.2 批次 B（任务 + 线索）

页面：TASK-01~05、CLUE-01。

验收要点：
1. 任务状态由服务端状态机驱动，客户端不做越权流转。
2. 实时事件按 `event_id` 去重，按 `version` 防乱序。
3. WS 不可用时可降级 `events/poll`，并在 UI 标识降级状态。

### 7.3 批次 C（家庭域）

页面：HOME-01、PAT-01~03、GUA-01、TAG-01。

验收要点：
1. 围栏参数校验符合接口规则，异常码有明确提示。
2. 监护高风险操作必须二次确认。
3. 标签状态变化后详情与历史同步刷新。

### 7.4 批次 D（运营 + AI + 管理）

页面：ORD-01~05、AI-01~04、ADM-01~08。

验收要点：
1. 物资工单按角色和状态机控制按钮可见性。
2. AI 对话支持流式渲染与降级提示（`E_AI_4292`、`E_AI_4293`）。
3. 管理域高风险操作必须审计留痕与二次确认。

## 8. 质量门禁与发布门禁

### 8.1 CI 必过项

1. `./gradlew clean test`
2. `./gradlew detekt ktlintCheck`
3. `./gradlew assembleDebug`
4. 关键模块 instrumentation tests（夜间任务可接受）

### 8.2 强制回归用例

1. ID 全字符串传输。
2. 写请求 `X-Request-Id` 注入。
3. 429 + `Retry-After` 倒计时。
4. 匿名凭据失效链路。
5. WS 乱序去重与防覆盖。

### 8.3 发布策略

1. 内测 -> 灰度 -> 全量。
2. 灰度观测 24 小时：崩溃率、ANR、接口失败率。
3. 发现高危问题时通过开关关闭 AI 或实时推送入口。

## 9. 团队分工建议

1. 平台组：多模块、网络、会话、安全存储、实时链路。
2. 业务组 A：匿名、鉴权、通知、任务。
3. 业务组 B：患者、监护、标签、物资、AI、管理。
4. 测试组：契约回归、关键 UI 自动化、发布门禁把关。

## 10. 第一周启动排程（可直接执行）

### Day 1

1. 完成 P0 契约冻结会议。
2. 产出四维矩阵初版（需求/页面/接口/数据）。
3. 输出高风险清单与确认责任人。

### Day 2

1. 完成架构脚手架方案评审。
2. 输出模块边界和依赖规则。
3. 建立联调问题单模板。

### Day 3

1. 落地 `core-network` 规范。
2. 落地统一响应解析与错误映射。
3. 打通 trace_id 日志贯通。

### Day 4

1. 跑通登录与会话最小闭环。
2. 实现 401 失效统一处理。
3. 补齐对应单元测试与集成测试。

### Day 5

1. 跑通通知列表与已读回滚策略。
2. 组织首轮后端联调演示。
3. 评审并冻结下一周开发包。

## 11. 风险与预案

| 风险 | 触发信号 | 预案 |
| :--- | :--- | :--- |
| 接口频繁变更 | 每周变更条目持续上升 | 设冻结窗口，变更先文档后开发 |
| 38 页面并行导致集成冲突 | 合并冲突、联调失败集中出现 | 按闭环批次交付，不做大爆炸集成 |
| 实时链路不稳定 | WS 断连率高、事件丢失 | 启用轮询降级 + 增量补洞机制 |
| 匿名链路风控漏实现 | 429/4012 工单增多 | 将 429/4012 设为阻断验收项 |
| 发布前质量回归不足 | 缺陷在灰度后集中出现 | 强制契约回归日 + 发布前冻结期 |

---

## 12. 完成定义（DoD）

1. 代码完成并通过 Code Review。
2. 与后端联调通过，关键错误码处理完整。
3. 单测、集成测试、关键 UI 测试通过。
4. 可观测性到位：日志、埋点、trace_id 可追踪。
5. 文档同步更新：接口、状态机、错误码变更记录齐全。
6. 灰度指标与回滚预案可执行。

---

## 13. 执行检查清单（每周）

1. [ ] 本周新增接口是否完成契约评审。
2. [ ] 写请求是否全部具备幂等请求号。
3. [ ] 401/429/业务码是否具备明确用户反馈。
4. [ ] WS 去重与防乱序规则是否覆盖自动化测试。
5. [ ] 新页面是否满足四态（Loading/Empty/Error/Content）。
6. [ ] 页面埋点是否覆盖进入、成功、失败、重试、退出。
7. [ ] 发布门禁是否全部通过。
